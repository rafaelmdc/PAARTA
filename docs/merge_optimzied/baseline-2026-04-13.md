## Query Shape Notes

Observed request patterns:

- merged protein and merged repeat-call list pages do not execute many queries,
  but each path still performs one large `SELECT` from `browser_repeatcall`
  with related joins and then groups the entire evidence set in Python
- accession list still combines grouped genome queries with a materialized
  repeat-call evidence pass and extra `COUNT(*)` queries
- accession detail has only two SQL queries on this small run, but it is the
  slowest measured path because it loads the full accession evidence set and
  expands the merged summary inline
- taxon detail in merged mode is still composition-heavy:
  - branch descendant count
  - accession group counts
  - merged protein grouping
  - merged residue grouping

The main failure mode is therefore confirmed:

- merged mode is not dominated by query count alone
- merged mode is dominated by full raw-evidence materialization and Python-side
  grouping before pagination or rendering

## Current Bottlenecks

The current merged hot paths still have these structural problems:

1. Merged proteins and merged repeat-call lists group raw `RepeatCall` rows in
   Python before pagination.
2. Accession analytics and accession detail still compute merged counters from
   raw evidence instead of a serving layer.
3. Taxon-detail merged counts still call helper paths that fully group the
   active evidence scope.
4. First-page merged templates still expand a large amount of provenance inline
   instead of serving a bounded preview.

## Conclusion

This baseline is sufficient to proceed with the persisted merged serving-layer
design.

The next useful implementation step is not more query trimming on the current
helpers. The next useful step is to populate the new merged summary and
occurrence tables during import and then move the merged read paths onto those
tables.
